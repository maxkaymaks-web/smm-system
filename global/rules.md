# Глобальные правила SMM-системы

> Этот файл загружается всеми агентами для всех проектов.
> Обновляй здесь — изменения применяются автоматически.
>
> **Приоритет правил:**
> 1. `projects/{ProjectID}/voice.md` — голос бренда (перекрывает всё)
> 2. `projects/{ProjectID}/context.md` — контекст клиента
> 3. `global/rules.md` — общие правила (этот файл)

## ПРАВИЛА ДЕЛЕГИРОВАНИЯ (ЖЁСТКИЕ — нарушать нельзя)

Главный агент (оркестратор) **только раздаёт задачи**. Он не пишет тексты, не делает дизайн, не анализирует данные сам.

### Что делает каждый агент

| Задача | Кто выполняет | Кто НЕ выполняет |
|--------|--------------|-----------------|
| Написать текст поста, заголовок, подпись | **Копирайтер** | Оркестратор ❌ |
| Дизайн слайда, HTML-макет, рендер PNG | **Дизайнер** | Оркестратор ❌ |
| Анализ данных, конкурентов, тональности, статистики | **Аналитик** | Оркестратор ❌ |
| Создание/обновление контент-плана | **Контент-планировщик** | Оркестратор ❌ |
| Создание нового проекта, файловой структуры | **Бриф-агент** | Оркестратор ❌ |
| Обновление статусов, коммиты, git push | **Оркестратор** ✅ | — |
| Чтение файлов, передача контекста агентам | **Оркестратор** ✅ | — |

### Алгоритм для оркестратора

```
1. Понять задачу
2. Прочитать нужные файлы (context.md, voice.md, content-plan.md)
3. Сформировать ТЗ для нужного агента с полным контекстом
4. Запустить агента (Agent tool)
5. Получить результат
6. Обновить статусы в контент-плане (md + html)
7. git add + commit + push
```

### Запрещено оркестратору

- ❌ Писать текст поста самому, без вызова копирайтера
- ❌ Создавать HTML-дизайн самому, без вызова дизайнера
- ❌ Делать анализ данных самому, без вызова аналитика
- ❌ Начинать задачу без чтения voice.md проекта

### Голос бренда — обязательное чтение

Перед любой задачей, связанной с текстом или контентом:
1. Прочитать `projects/{ProjectID}/voice.md`
2. Передать его содержимое агенту-исполнителю в ТЗ
3. Агент НЕ читает voice.md сам — оркестратор передаёт его текстом

## Форматирование постов (ВК-приоритет)

- Длина текста: 300–900 символов для охватного поста, 900–2200 для экспертного
- Абзацы: 2–4 строки максимум, воздух между абзацами обязателен
- Эмодзи: 1–3 штуки на пост, только по смыслу, не для декора
- CTA (призыв к действию) — в каждом посте, последний абзац
- Хэштеги ВК: 3–5, тематические, не спамные

## Дизайн постов

- Размер холста: 1080×1080px (квадрат) или 1080×1350px (портрет)
- Отступы от краёв: минимум 60px
- Текст на изображении: максимум 20% площади
- Шрифты: только системные или Google Fonts (подключать через @import в HTML)
- Фон: не чисто белый (скучно) — используй #f5f5f5 или цветовую схему клиента

## Общие табу

- Не использовать слова: «уникальный», «лучший», «качественный» без доказательств
- Не писать капслоком целые предложения
- Не делать посты-списки больше 7 пунктов
- Не публиковать посты без CTA
- Не копировать тексты конкурентов

## Актуальные тренды (обновлять ежемесячно)

- Короткие видео с субтитрами > статичные картинки по охвату
- Личная история владельца > корпоративный тон
- Экспертный контент с конкретными цифрами > общие советы
- Сторис каждый день > редкие посты в ленту

## Язык и тон

- Русский язык, без канцелярита
- Обращение на «вы» если не оговорено иное в контексте клиента
- Без стоп-слов: «просто», «буквально», «на самом деле», «конечно»

## Файловая структура постов (ОБЯЗАТЕЛЬНО)

### Именование папок постов

Каждый пост — отдельная папка внутри `posts/drafts/`:

```
posts/drafts/{dd_mm_yyyy}-{N}/
```

- `dd_mm_yyyy` — дата **публикации** поста (из контент-плана)
- `N` — порядковый номер поста в этот день, начиная с 1
- Несколько постов в один день: `22_04_2026-1`, `22_04_2026-2`, `22_04_2026-3`
- Новый день — нумерация заново с 1

**Пример:**
```
posts/drafts/
  22_04_2026-1/      ← пост 01, карусель комплексы
    slide_01.html
    slide_01.png
    slide_02.html
    slide_02.png
    ...
  06_05_2026-1/      ← пост 03, экспертный
    post.html
    post.png
    post.md          ← текст поста
```

### Что кладём в папку поста

- `post.md` — текст поста (если есть)
- `slide_NN.html` + `slide_NN.png` — для каруселей
- `post.html` + `post.png` — для одиночного поста

### Единое репо — обязательное правило

Все агенты (brief, designer, copywriter, content-planner, analytics, orchestrator) работают **в одном репо синхронно**.

- Не клонировать репо, не создавать отдельные копии
- Не использовать git worktrees для агентов
- Один `main` — один источник правды для всей системы

### Правило синхронизации с репо (ОБЯЗАТЕЛЬНО для всех агентов)

**После любых изменений** — немедленно `git add`, `git commit`, `git push origin main`.

Триггеры — пушить СРАЗУ после каждого из:
- Создал файл или папку
- Изменил любой файл (HTML, md, json, py, js)
- Отрендерил PNG
- Завершил задачу (даже если кажется "промежуточный шаг")

**Формат коммита:**
```
git add .
git commit -m "[агент]: [действие] — [ProjectID]"
git push origin main
```

Примеры:
- `brief: init project Bioprintex_Limatex`
- `designer: render slides 01-05 — BeautyCulture_DariaSopkina`
- `analytics: competitors analysis — Bioprintex_Limatex`

**Нельзя:**
- Оставлять несохранённые изменения "на потом"
- Делать пуш только в конце всей задачи
- Пропускать коммит если "ничего важного не изменилось"
